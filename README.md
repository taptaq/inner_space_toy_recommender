# Inner Space Gear Navigator

一个以“内太空探索”为主题的亲密装备智能推荐项目。

它不是一个普通商品列表，而是一个更像“私密选择导航仪”的体验：用户不用一上来就面对密密麻麻的型号、参数和营销词，而是通过一段轻量问答，逐步明确自己的预算、使用场景、隐私需求、体验偏好和关系状态，最后获得更适合自己的装备推荐、避坑提醒和知识解释。

如果要用一句话介绍：

> 帮用户在不尴尬、不迷路、不被参数绕晕的情况下，找到更适合自己的亲密装备。

## 这个项目想解决什么

很多人在选择亲密装备时并不是没有商品可看，而是信息太杂、参数太多、羞于提问，也不知道自己真正应该看什么。

这个项目围绕几个真实困扰来设计：

- 不知道第一台应该怎么选。
- 看不懂强度、静音、防水、材质、形态这些参数。
- 担心买回来不适合自己或伴侣。
- 在意隐私、收纳、声音、清洁和同住环境。
- 想要更科学一点的解释，而不是只看商家宣传。
- 想保存一次匹配结果，之后继续比较或复盘。

所以它的核心不是“卖更多商品”，而是降低选择焦虑，帮用户完成一次更安心、更有依据的决策。

## 功能模块

### 1. 智能匹配问答

用户从首页点击“开始匹配”后，会进入一组围绕真实使用场景设计的问题。

问答不是简单收集标签，而是尽量模拟真实购买前会纠结的点，例如预算范围、是否新手、是否情侣共玩、是否同住、是否需要静音、对强度是否确定、是否更在意隐蔽性或陪伴感等。

前面的选择会影响后续题目的侧重点。比如单人探索、情侣共玩、新手第一台、隐私敏感场景，背后的推荐逻辑都会有所不同。

### 2. 场景化加载页

匹配过程中不是空白等待，而是以深空碎片的方式展示一些简短知识点。

这些碎片像漂浮在页面周围的太空残片，不遮挡核心内容，也不会像传统知识卡片那样打断流程。它的作用是让用户在等待时顺手获得一点有用信息，比如静音、防水、材质、清洁或新手选择相关的小提示。

### 3. 匹配结果页

结果页会给出推荐装备，而不是只丢出一排商品。

每个推荐都会尽量回答几个问题：

- 为什么它适合当前用户。
- 它对应哪些偏好条件。
- 可能有哪些不适合的地方。
- 如果不喜欢这个方向，可以考虑什么替代路线。
- Top 推荐之间有什么差异。

这样用户看到的不是“AI 说买这个”，而是能理解推荐背后的原因。

### 4. 快速微调结果

结果页支持轻量微调，比如：

- 更安静一点。
- 预算低一点。
- 更适合新手。

点击微调后会在保留当前问卷主干的前提下重新计算推荐，并展示 loading 和提示文案。同一轮结果里已经应用过的微调方向会被禁用，避免用户重复点击同一个调整项；但重新开始新问卷或回到题目修改条件时，不会把旧微调状态错误带回去。

这个功能比“重新答一遍问卷”更自然，因为很多真实决策都是在看到结果后才发现：我其实想要再安静一点、再便宜一点、或者再适合新手一点。

### 5. 我的装备匹配档案

用户登录后可以保存当前匹配方案，形成“我的装备匹配档案”。

档案里会记录当时的推荐结果、偏好条件、筛选标签和主要结论。之后用户可以回来查看，不用每次重新答题，也方便在不同设备上继续比较。

这部分不是只存在本地，因为移动端和跨设备场景需要同步；但偏好、条件和推荐结果这类相对私密的信息会加密后再保存到数据库。

### 6. 装备库

装备库用于承接更明确的浏览需求。

用户可以在这里查看装备参数、品牌、价格、来源链接、适用场景和原始描述等信息。它更像是一个补充资料库：当用户已经大概知道自己想看什么时，可以进一步筛选和比较。

当前装备库筛选已经不是只有基础性别过滤，而是包含一套联动分类体系：

- 性别 / 适用对象
- 类型
- 类型细分
- 价格区间
- 静音阈值
- 品牌、材质、来源等扩展条件

其中“类型”和“类型细分”会和“适用对象”联动。比如男性向不会展示纯女性类别，通用类会展示适合通用方向的分类。空类型会按“其他”处理，方便前端保持筛选完整性。

在数据清洗上，项目会保留润滑液、避孕套、内衣等护理周边类目，但会把配件、连接器、适配器等无关项从推荐库中剔除，避免污染主筛选结果。

相比首页直接把装备库作为主入口，目前更推荐把它作为次级入口，因为大多数用户真正要完成的任务不是“逛数据库”，而是“帮我选”。

### 7. 知识星云

知识星云是项目里的知识库入口，但它不是传统文章列表。

首页展示 5 个像星云雾团一样的主题区域，点击后进入对应主题详情。详情页用飞船驾驶舱的形式呈现知识卡片，让每张卡片像驾驶舱屏幕上的参数模块，用户可以点开查看完整内容。

当前 5 个主题是：

- 参数与体验原理：理解参数、结构和体感之间的关系，避免被营销词带偏。
- 人群与场景导航：根据不同人群、关系和使用环境，找到更贴合的选择方向。
- 新手第一台：帮助第一次选择的人降低试错成本。
- 伴侣互动与边界：关注情侣共玩、沟通、边界和共同体验。
- 清洁收纳与长期维护：处理清洁、防水、收纳、续航和长期使用习惯。

每张知识卡片都可以记录热度。同一个用户重复点击同一张卡片不会重复计数，所以热度更接近“有多少人对这个内容感兴趣”，而不是单纯点击次数堆叠。

### 8. 关联知识推荐

知识详情页会展示可能关联的参数卡片。

后续内容越来越多时，不适合一次性把所有卡片都堆出来。因此项目会根据卡片内容、主题、标签和 embedding 相似度，给用户推荐更可能相关的下一组知识。

这样知识库不是静态目录，而更像一张可以探索的亲密装备知识地图。

### 9. 登录与隐私

产品交互上使用“用户名 + 密码”，不强制用户填写真实邮箱。

用户可以登录保存匹配档案，同时项目会把敏感档案内容加密存储。这样既能支持移动端和跨设备同步，也尽量减少私密偏好明文暴露的风险。

隐私表达是这个项目很重要的一部分，因为亲密装备选择本身就带有天然的尴尬感和安全感需求。

## 当前推荐链路

### 1. 初次出结果

当前结果不是直接把数据库商品原样丢给用户，而是先经过本地结构化候选筛选，再进入 AI 重排与增强：

- 本地根据预算、静音、防水、结构形态、适用对象等规则筛出候选池
- AI 对候选池进行 Top3 重排并生成主推荐理由
- AI 再为备选卡片补充简短说明，并生成购买建议

因此结果页上的推荐是“结构化规则 + AI 理由增强”的组合，而不是单纯文案生成。

### 2. 重新生成推荐

结果页的“重新生成推荐”走的是专门的重校准链路，不是简单重复同一次请求。

当前逻辑是：

- 每次点击“重新生成推荐”，会重新构建当前候选池与上下文
- 只轮换 `Top3` 重排阶段的首发模型，让用户获得不同模型视角下的排序结果
- 备选说明与购物建议阶段仍保持更稳定的优先链路
- 如果当前首发模型失败，会继续按兜底顺序自动尝试后续 provider

也就是说，它现在更接近“换一个模型重新审视主推荐，再补充说明”，而不是“把整条链路无差别全部换掉”。

### 3. 当前模型与兜底

当前 DMXAPI / 官方 provider 体系已经扩展为：

- `dmxapi-mimo`
- `dmxapi-minimax`
- `dmxapi-qwen`
- `dmxapi-glm`
- `dmxapi-kimi`
- `dmxapi-claude`
- `dmxapi-gemini`
- `dmxapi-grok`
- `dmxapi-gpt`
- `dmxapi-kimi-k2`
- `deepseek`
- `qwen`
- `glm`

其中：

- `dmxapi-qwen` 当前使用 `qwen3.5-27b`
- DMXAPI 扩展兜底中已加入 `claude-opus-4-7`、`gemini-3.1-pro-preview-ssvip`、`grok-4.2-nothinking`、`gpt-5.4`、`kimi-k2.6`

这套链路的目标不是让用户手动选模型，而是把模型切换隐藏在“重新生成推荐”这个产品动作后面。

## 体验风格

整个项目走的是“深空探索 / 星云漫游 / 飞船驾驶舱”的视觉方向。

它希望把一个可能尴尬、紧张、信息过载的选择过程，变成一段更轻、更安全、更有探索感的体验：

- 首页像进入一段内太空任务。
- 答题页用星空背景承接探索感。
- 加载页用漂浮知识碎片降低等待感。
- 结果页像生成一份装备导航报告。
- 知识星云像进入一片可探索的知识宇宙。
- 知识详情页像坐在驾驶舱里查看参数屏幕。

这个视觉主题不是单纯装饰，而是服务用户心理：让“私密选择”变得不那么直白、不那么压迫，也更容易开始。

## 项目运行

以下内容主要给开发和维护使用。

项目要求 Node.js `20.19.0`。

```bash
npm install
npm run server
npm run dev
```

默认端口：

- 前端 Vite：`http://localhost:3009`
- 后端 Express：`http://localhost:3010`
- Vite 会把 `/api` 代理到 `http://localhost:3010`

### Vercel 部署说明

线上环境现在采用“前端静态产物 + 单入口 Vercel Function”：

- 本地开发仍然使用 `src/server/index.ts` 启动 Express
- 线上部署时由 `api/index.ts` 承接所有 `/api/*` 请求
- `vercel.json` 会把 `/api/:path*` 重写到 `/api`，再由函数内还原原始 Express 路径

这样前端请求地址不需要改动，开发和线上都继续使用同一套 `/api/*` 路由。

也可以一条命令同时启动：

```bash
npm run dev:all
```

如果遇到 `EADDRINUSE: address already in use 0.0.0.0:3010`，说明后端端口被占用，可以先查看占用进程：

```bash
lsof -nP -iTCP:3010 -sTCP:LISTEN
```

## 维护注意事项

- 普通用户 UI 不应该暴露知识卡片编辑入口；新增/编辑、featured 等运营能力后续应挂到管理员权限后再开放。
- 推荐逻辑会同时使用商品结构化字段和 `raw_description` / 标签信号，否则只靠价格、性别、材质、噪音等字段容易丢失细节。
- 当前运行时推荐库只使用 `public.recommender_toys`，不再把 `recommender_items` 作为有效业务表。
- `/api/recommender/toys` 会优先读取 `safe_display_name`，并在前端统一使用安全展示名，避免原始名称里出现不适合直接展示的内容。
- 装备库类型与类型细分已经回填进 `recommender_toys.type_code` / `subtype_code`，并和性别筛选联动。
- 图片展示允许真实远程图、`data:image/...`、`blob:` 等图片源；如果没有可用图，会退回默认渐变占位图，而不是空白黑块。
- “重新生成推荐”现在会轮换 Top3 重排首发模型，但保留自动兜底；结果页对外仍应表现为统一的产品动作，而不是让用户手动理解 provider 细节。
- README 前半部分偏产品介绍，适合用于对外宣传；后半部分保留开发运行信息，方便继续维护。
