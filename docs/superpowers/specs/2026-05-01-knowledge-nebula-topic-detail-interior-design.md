# 知识星云详情页内核重设计 Spec

日期：2026-05-01

## 背景

当前 `/knowledge/:slug` 详情页虽然已经从普通知识卡片页过渡到较强的深色星空风格，但用户对现阶段效果仍然不满意，问题集中在三个方面：

- 详情结构仍然像 `Hero + 卡片列表`，没有真正形成“进入某团星云内部”的沉浸感
- 知识卡片一旦强化视觉权重，容易再次出现遮挡、堆叠、留白失衡和“塑料感”
- 点击后的知识弹窗承载信息偏少，更像轻量提示层，不像真正的知识详情空间

因此，这次不是继续微调卡片边框、发光、hover 或文案，而是重新定义知识星云详情页的体验骨架：用户点击主题后，不是进入普通详情页，而是像镜头穿入一团主题星云内部，在云层中探索分散漂浮的知识节点，并在点击节点后进入独立的深空信息层。

## 本次目标

- 把详情页从“普通详情布局”升级为“进入主题星云内部”的沉浸式场景
- 保留主题标题和一句摘要，但不再使用传统 Hero 面板作为核心视觉
- 让知识内容以 `分散漂浮的节点` 形式围绕主题云核分布
- 节点之间要有空间层次和关联感，但不能再次变成混乱堆叠
- 点击节点后进入 `独立深空信息层`，承载更完整的知识内容、来源和同主题延展入口
- 整体风格追求高级深空摄影感，避免玻璃塑料感、廉价赛博感和过度 UI 化
- PC 与移动端都保留一致的世界观和视觉语言

## 已确认的用户决策

- 详情方向：整体做成“进入某团星云内部”的形态
- 视觉层级：`3D 星云场景 + 2D 信息层`
- 节点布局：知识节点自由分散漂浮，不走规则卡片网格
- 弹出层策略：点击节点后进入独立的深空信息层，不做原地展开
- 中心区策略：保留主题标题和一句摘要，作为主题星云核心
- 风格要求：高级、精致、克制，不要塑料感，可以用 3D 能力做空间层次

## 核心体验定义

用户进入某个知识主题详情页时，应当感觉自己正在穿过星云表层，进入这团主题星云的内部。

画面的中心是一个具有体积感和轻微呼吸感的 `主题云核`。标题和摘要不再悬在普通面板上，而是嵌在这团云核的低亮信息层中。用户的注意力先被中心云核稳定住，然后再自然地看到四周漂浮的知识节点。

这些节点不是普通矩形卡片，而是附着在云层边缘或云带中的信息碎片。它们在空间上分散、有远近、有前后层次，像被这团主题星云的引力悬住。用户点击任意节点后，当前节点被抽离、拉近，并进入独立的深空信息层；背景星云仍然存在，只是退到后景，从而保持“我还在这团星云里”的连续感。

整套体验应当是：

- 进入主题星云
- 在云层中探索分散知识节点
- 点开节点进入独立深空信息层
- 关闭后回到原来的星云内部位置

## 视觉构图

### 1. 主题云核

主题云核是详情页的中心视觉锚点。

构图要求：

- 位于页面中央偏上的稳定区域
- 保留 `标题 + 一句摘要`
- 不再是完整矩形 Hero 卡片
- 以云核、雾层、局部光晕和低亮文字托底形成信息核心
- 文字始终清晰，不允许被云层或粒子遮挡

视觉气质：

- 深空摄影感，而不是赛博玻璃感
- 暗底、冷青蓝、银白高光
- 发光收敛、柔和、非塑料质感
- 中心亮度明确，但边缘不过分发白

### 2. 知识节点

知识节点不再是规则卡片，而是“附着在云团边缘的信息碎片云块”。

形态要求：

- 节点外轮廓轻微不规则
- 边缘有克制的亮边和星尘脱落感
- 中央文字承载区必须稳定、清晰、不模糊
- 节点之间体积不一致，大小至少分为三级

建议结构：

- 1 个主节点
- 2 至 3 个中节点
- 3 至 4 个小节点

视觉原则：

- 先是“云团碎片”，然后才是“信息块”
- 不要再出现统一矩形、统一发光、统一厚边框
- 不要依赖大量 blur 来制造氛围，否则会再次导致文字不清和廉价感

### 3. 背景环境

背景不是纯黑底，也不是简单星点平铺。

需要包含：

- 远景微弱星点
- 中景局部云带
- 前景极少量漂浮星尘
- 与当前主题 accent 一致的局部色场

环境层的使命不是抢戏，而是让用户相信当前页面确实位于某团星云内部。

## 空间分布规则

### 1. 安全区

中心主题云核周围必须留出 `安全半径`。

规则：

- 节点不得压入中心安全区
- 安全区确保标题、摘要和中心云核始终完整可读
- 安全区随 viewport 自适应，移动端会更保守

### 2. 引力环带

外围节点不是随机撒开，而是沿着若干 `引力环带` 分布。

规则：

- 近景节点更少、更大
- 中景节点数量适中，是主阅读层
- 远景节点更轻、更小，用于拉开空间层次
- 左右上下都要被占用，避免再次挤成单侧堆叠

### 3. 关联性表达

节点之间需要有同属一个主题的“相关感”，但不做明显流程图或强连线。

表达方式建议：

- 极淡的粒子牵引线
- 局部星尘流向一致
- hover/selected 时周边节点轻微响应

关联性要被感知到，但不能让画面变成理工科星图。

## 动效与状态节奏

### `enter`

进入详情页时，镜头从主题外层推入主题星云内部。

表现：

- 云核亮度上升
- 外围节点逐步显形
- 粒子密度从低到中
- 相机停止后页面进入可探索状态

### `idle`

空闲态下，整个详情页处于缓慢呼吸和空间漂移状态。

表现：

- 云核缓慢脉动
- 节点低频漂浮
- 前后景粒子极轻移动
- 不同节点不应统一频率，避免机械感

### `hover`

hover 到知识节点时，不是简单放大，而是节点所在小云团被“扰动”。

表现：

- 边缘星尘轻微荡开
- 节点亮度提升
- 文本承载区更清晰
- 当前节点向前轻微浮起

### `open`

点击节点后，节点先被拉近，再转化为独立深空信息层。

表现：

- 背景星云轻微后退、失焦或暗下去
- 当前节点扩散并抽离
- 信息层成形时保留来自原节点的运动连续性

### `close`

关闭信息层时，信息层缩回到原来的节点位置，而不是直接消失。

这样才能形成“从星云中取出一块知识核心，再放回去”的连贯体验。

## 独立深空信息层

这层不做普通 modal，而是做成“从当前节点抽离出的深空舱室”。

### 结构

分为三段：

1. 顶部信息层
2. 中部正文层
3. 底部延展层

### 顶部信息层

包含：

- 节点标题
- 一句结论或摘要
- 主题归属
- 重点状态
- 标签

这里负责让用户快速知道“我打开了什么”“它属于哪一类内容”。

### 中部正文层

包含：

- 更完整的正文段落
- 清晰的分段与节奏
- 可滚动阅读

要求：

- 比当前弹层承载更多正文
- 可读性优先于视觉炫技
- 不做过重的透明/模糊叠加

### 底部延展层

包含：

- 来源链接
- 同主题其他碎片入口
- 必要时的相关标签或补充状态

这部分负责增强知识深度和节点之间的跳转关联。

### 视觉要求

- 不做常规玻璃弹窗
- 边缘是非常薄的发光轮廓
- 内部是深色渐变、少量漂浮粒子和稳定内容层
- 保持“独立深空舱室”感，而不是后台浮层感

## 技术方案

### 选型

继续沿用当前已确认的技术栈：

- `three`
- `@react-three/fiber`
- `@react-three/drei`
- DOM overlay 作为文字与点击层

### 实现原则

采用 `3D 环境 + 2D 信息节点` 的混合方案，而不是全 DOM 模拟，也不是全 3D 文本交互。

原因：

- 3D 负责真实空间层次、景深、云核和星尘
- DOM 负责文字清晰度、点击热区和弹层可维护性
- 能同时保证高级感与阅读效率

### 组件职责建议

#### `KnowledgeNebulaPage`

负责：

- 详情页路由状态
- 进入/退出节奏
- 当前主题与当前节点选择

#### `NebulaScene3D`

负责：

- 中心主题云核
- 远中近景星尘
- 背景云层
- 空间相机层次

#### `NebulaNodeLayer` 或等价 DOM/Overlay 层

负责：

- 节点文字
- 节点热区
- hover 与 click 反馈
- 移动端可读性保底

#### `KnowledgeNebulaTopicSections`

当前需要从“卡片列表”角色，重构为：

- 节点信息描述层
- 当前节点详情层
- 同主题关联跳转层

它不应再继续维持普通 grid 卡片布局为主的实现方向。

## PC 与移动端策略

### PC

- 保留更完整的前后景层次
- 节点数量和景深更丰富
- hover 扰动与轻微视差可更明显

### 移动端

- 节点数量可略减，但不可退回成普通列表页
- 中心安全区更大
- 节点默认透明度与文字对比更高
- 降低粒子和实时扰动密度

移动端是同一世界观下的收敛版本，不是另一套普通 UI。

## 非目标

本次不包含：

- 改写知识内容的数据结构
- 重做知识首页 Hub
- 引入高成本体积 raymarching
- 纯追求写实天文模拟
- 自动提交代码或设计文档

## 成功标准

如果这次详情页重设计成功，用户进入某个主题后应当立即感受到：

- 这不是普通详情页，而是真的进入了一团主题星云内部
- 中心有稳定的主题核心，四周节点在漂浮，但不乱
- 节点之间存在空间关系和关联性，而不是一堆卡片
- 点击节点后的信息层更完整、更高级、更像深空知识舱室
- 整体观感高级、克制、有层次，不再有廉价塑料感

## 实施建议

1. 先重构详情页整体场景骨架，去掉 `Hero + grid list` 的主结构
2. 再定义中心云核与节点安全区的空间规则
3. 然后重做节点 overlay 与 hover/open 状态
4. 最后重构独立深空信息层，补齐更多信息承载和同主题跳转
