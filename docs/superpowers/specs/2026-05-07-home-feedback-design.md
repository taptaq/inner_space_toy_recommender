# 首页意见反馈功能设计

## 目标

在首页增加一个轻量、不打断主流程的意见反馈入口，让任何访客都可以匿名提交文本反馈，并可选附带最多 3 张截图，数据直接写入现有 PostgreSQL 数据库。

## 范围

本次仅实现首页反馈能力，不扩展到结果页、装备库或知识星云页。

包含内容：

- 首页新增 `意见反馈` 入口
- 站内模态弹窗表单
- 匿名提交
- 文本反馈必填
- 截图上传可选，最多 3 张
- 新增后端接口写入数据库
- 新增数据库表保存反馈记录
- 基本前后端校验与提交状态提示

不包含内容：

- 管理后台
- 反馈列表查询页
- 邮件通知、IM 通知
- 独立对象存储
- 登录态绑定用户身份

## 用户体验

### 首页入口

在首页主卡片底部，和“先随便看看装备库”“看看知识星云”同层级增加一个次级入口按钮 `意见反馈`。视觉风格沿用现有首页次级入口体系，保持低打扰，不抢占“开始匹配”的主动作。

### 反馈弹窗

点击后打开站内模态框，不跳页。弹窗内容保持简短：

- 标题：`意见反馈`
- 说明文案：鼓励用户反馈问题、体验卡点或建议
- `反馈内容` 多行输入框，必填
- `截图上传` 区域，可选，最多 3 张
- 已选截图缩略预览
- 每张截图支持移除
- 底部按钮：`提交反馈`、`暂不反馈`

### 提交反馈

提交时展示加载状态，防止重复提交。

成功后：

- 给出明确成功提示
- 清空当前表单状态
- 自动关闭弹窗或短暂停留后关闭

失败时：

- 保留已填写文本和已选截图
- 给出简短错误提示，支持重试

## 数据模型

新增表：`public.user_feedback`

建议字段：

- `id uuid primary key default gen_random_uuid()`
- `message text not null`
- `screenshots jsonb not null default '[]'::jsonb`
- `page_route text not null default '/'`
- `user_agent text`
- `created_at timestamptz not null default now()`

其中：

- `message` 保存用户反馈正文
- `screenshots` 保存截图数组，每项为压缩后的 data URL 字符串
- `page_route` 记录提交来源，本次固定来自首页 `/`
- `user_agent` 便于后续粗略排查环境问题

不保存用户账号信息，不保存额外敏感身份信息。

## 前端实现

### 页面结构

在 `HomePage` 中新增一个反馈入口按钮，并新增反馈弹窗组件，组件可先放在 `src/components` 下独立维护，避免首页文件继续膨胀。

### 表单状态

前端管理以下状态：

- 弹窗开关
- 反馈内容
- 已选截图列表
- 提交中状态
- 成功/失败提示

### 截图处理

前端允许用户选择图片文件，限制为常见图片类型：

- `image/png`
- `image/jpeg`
- `image/webp`

上传规则：

- 最多 3 张
- 超出数量时立即阻止并提示
- 单张文件过大时在前端阻止
- 提交前可做浏览器侧压缩，再转为 data URL

为了控制数据库体积，本次实现建议在浏览器侧将截图压缩为较小分辨率与质量后再提交。

## 后端实现

### API

新增接口：

- `POST /api/feedback`

请求体：

- `message: string`
- `screenshots: string[]`
- `pageRoute: string`

接口行为：

- 不要求登录
- 校验 `message` 非空
- 校验 `screenshots.length <= 3`
- 校验截图字段均为图片 data URL 字符串
- 写入 `public.user_feedback`
- 成功返回 `201`

### 服务组织

沿用现有 `Express + pg Pool` 模式，新增：

- feedback store
- feedback route handler
- schema ensure 函数

并在 `src/server/index.ts` 中完成挂载与初始化。

## 校验规则

### 前端校验

- 反馈内容去首尾空格后不能为空
- 截图最多 3 张
- 只接受图片类型
- 不满足条件时禁止提交

### 后端校验

- `message` 为空返回 `400`
- 截图数量大于 3 返回 `400`
- 非法截图数据返回 `400`
- 数据库异常返回 `500`

后端校验不能依赖前端，必须独立生效。

## 测试策略

### 前端测试

- 首页存在 `意见反馈` 入口
- 弹窗包含 `反馈内容` 与 `截图上传`
- 文本为空时不能提交
- 选图超过 3 张时出现限制提示
- 提交中状态能正确展示

### 后端测试

- schema ensure 会创建 `user_feedback` 表
- 匿名反馈可成功写入
- 空 message 会被拒绝
- 超过 3 张截图会被拒绝

## 非目标与后续扩展

本次不做运营后台与通知链路。后续如需要，可以在不破坏当前结构的前提下继续扩展：

- 增加反馈管理页
- 增加状态字段，如 `pending / resolved`
- 增加截图改走对象存储
- 扩展到结果页、装备库、知识卡片页

## 实现原则

- 保持首页主任务“开始匹配”为最强入口
- 反馈能力尽量轻，不制造额外理解成本
- 匿名可用，避免把反馈门槛抬高
- 先解决可收集问题的能力，再考虑后台治理
