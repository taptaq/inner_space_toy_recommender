# 身体人格测试资料包

> 这份文档用于把当前项目中的“身体人格测试”功能交给其他模型继续讨论、优化或生成视觉方案。  
> 当前项目主题是“内太空装备智能选品向导”，整体语气应保持克制、治愈、私密、尊重边界，避免低俗、猎奇或过度直白。

## 1. 功能定位

身体人格测试是结果页中的一个延伸功能，用来回答：

- 当前推荐链路回答“你这次想要什么装备”。
- 身体人格测试回答“你长期更适合什么体验路线”。
- 测试结果需要能反向联动装备推荐，包括节奏、隐私、隐藏路线、收纳、噪音、外观伪装、可控性等维度。

当前入口文案：

- 模块名：`身体人格测试`
- 主标题：`看清你长期更适合哪类装备路线`
- 说明：`这次推荐回答你现在想要什么，身体人格会告诉你长期更适合什么节奏、隐私感和隐藏路线。`
- CTA：未测时 `开始测试`，已测后 `重新测一次`
- 付费提示：`免费先看基础画像，完整报告 0.5 元解锁。`

当前测试形式：

- 10 道题。
- 每题 3 个选项。
- 免费生成基础画像。
- 完整报告需要 0.5 元解锁。
- 目前支付链路是 mock/dev confirmation，后续可接入真实微信/支付宝等支付。

## 2. 当前用户流程

1. 用户完成装备推荐主流程，进入匹配结果页。
2. 结果页展示身体人格测试入口。
3. 用户点击开始测试，弹出 10 题测试弹窗。
4. 用户完成所有题目后点击 `生成我的身体人格结果`。
5. 前端本地先计算基础人格结果，同时把答题路径、候选产品池、基础结果写入后端。
6. 结果页展示免费基础画像。
7. 用户可以支付 0.5 元解锁完整报告。
8. 解锁后返回完整画像、隐藏路线总结、适配方向、暂不优先方向、人格匹配产品方向。
9. 若 AI 报告润色失败，系统使用本地确定性报告兜底。

## 5. 当前隐藏路线

当前隐藏路线用于表达“伪装感、低存在感、便携/收纳、外观安全感”等爽点。  
隐藏路线 code 也不建议轻易改动。

| code | 当前报告标签 | 解释方向 | 可能适合的视觉隐喻 |
| --- | --- | --- | --- |
| `zero_profile` | 无隐藏路线 | 用户并不强依赖伪装或隐藏，但可能重视安静、不被观察。 | 幽灵星系、低亮度星云、几乎透明的轨道 |
| `daily_object` | 日常器物型 | 表面越日常越安心，最好别人看见也不会联想到私密用途。 | 透镜星系、暗环、日用品轮廓但不出现具体产品 |
| `beauty_disguise` | 美貌伪装型 | 外观像漂亮小物件，有审美和反差的小彩蛋。 | 玫瑰星系 Arp 273、玻璃珐琅、精致星尘 |
| `pocket_ready` | 口袋随身型 | 不一定伪装，但必须小巧、好收、好带走。 | 小麦哲伦星云、口袋大小的星云碎片、紧凑轨道 |

隐藏强度：

- `hiddenPower >= 5`：`S`
- `hiddenPower >= 3`：`A`
- 其他：`B`

共居安心度：

- 与隐藏强度同步计算。
- `hiddenPower >= 5`：`high`
- `hiddenPower >= 3`：`medium`
- 其他：`low`

可继续讨论的问题：

- `zero_profile` 现在显示为“无隐藏路线”，是否会降低用户爽点？也许可以改成“低存在感型”。
- `daily_object` 现在是“日常器物型”，是否更适合叫“日常伪装型”。
- `beauty_disguise` 现在是“美貌伪装型”，是否更适合叫“精致伪装型”。
- 隐藏路线是否应该成为分享卡上的“隐藏副人格”。

## 6. 当前评分逻辑

当前算法是确定性加权：

1. 初始化 6 个主画像分数为 0。
2. 遍历用户选择的 10 个答案。
3. 每个答案对一个或多个主画像加权。
4. 每个答案可对一个或多个隐藏路线加权。
5. 主画像按分数排序，第一名为 `primaryPersonaCode`，第二名为 `secondaryPersonaCode`。
6. 隐藏路线按隐藏路线分数排序；如果总隐藏分为 0，则强制为 `zero_profile`。
7. 根据总隐藏分计算隐藏强度和共居安心度。

当前返回结果结构：

```ts
type BodyPersonaResult = {
  primaryPersonaCode: BodyPersonaCode;
  secondaryPersonaCode: BodyPersonaCode | null;
  hiddenRouteCode: HiddenRouteCode;
  hiddenPowerGrade: "S" | "A" | "B";
  coLivingComfortGrade: "high" | "medium" | "low";
  freeSummary: {
    title: string;
    blurb: string;
    why: string;
    hints: string[];
  };
};
```

当前算法优点：

- 可解释。
- 快。
- 不依赖 AI。
- 便于测试和回归。
- 适合做基础画像兜底。

当前算法限制：

- 没有根据用户答题组合生成更细腻的个性化表达。
- 主画像之间可能出现语义重叠。
- 隐藏路线主要来自少数题，粒度还可以继续细分。
- 目前还没有用户反馈闭环训练。
- 当前 `dimensionScores` 虽然入库，但前端传的是对象占位，尚未完整沉淀每个维度的详细分数。

## 7. 完整报告逻辑

完整报告由两层组成：

1. 本地确定性报告：`buildBodyPersonaFullReport`
2. AI 润色增强：`createBodyPersonaReportService`

完整报告结构：

```ts
type BodyPersonaFullReport = {
  title: string;
  portrait: string;
  hiddenRouteSummary: string;
  goodFits: string[];
  avoidNotes: string[];
  productPicks: Array<BodyPersonaCandidate & { personaScore: number }>;
};
```

候选产品结构：

```ts
type BodyPersonaCandidate = {
  id: string;
  name: string;
  score: number;
  tags?: string[];
  typeCode?: string | null;
  appearance?: string | null;
  maxDb?: number | null;
};
```

当前候选产品池：

- 来自结果页主推荐和备选产品。
- 当前取 `topProducts + backupProducts` 的前 8 个。
- 写入字段包括：`id`、`name`、`score`、`tags`、`typeCode`、`appearance`、`maxDb`。

当前产品重排规则：

- 对所有候选产品计算 `personaScore`。
- 基础分来自原推荐 `score`。
- 如果主画像是 `starlit_guard`：
  - `appearance === "high_disguise"` 加分。
  - `maxDb <= 45` 加分。
  - 标签包含 `伪装`、`静音`、`隐蔽` 加分。
- 如果主画像是 `comet_spark`：
  - 标签包含 `强刺激`、`直给`、`高能` 加分。
- 最终按 `personaScore`、原始 `score`、`id` 排序，取前 5 个。

当前完整报告文案仍偏基础，值得重点优化。

## 10. 视觉方向讨论：星系徽章与分享卡

之前生成的徽章图偏普通，主要问题是：

- 像通用 AI 星云图。
- 所有类型容易都变成“圆形星系 + 金属边框”。
- 缺少用户身份感。
- 缺少社交分享时一眼可懂的类型信息。

建议方向：

- 不建议纯 3D 游戏徽章，容易变成 NFT、游戏装备或廉价勋章。
- 不建议纯 2D 平面星图，记忆点不够。
- 更推荐 `2.5D 立体玻璃珐琅徽章 + 分享卡片`。

理想视觉关键词：

- 2.5D
- glass enamel
- semi-transparent cosmic core
- subtle raised relief
- delicate metallic rim
- collectible identity emblem

## 11. 可以交给其他模型继续讨论的问题

### 产品定位

- 身体人格测试更像 MBTI、星座测试、装备偏好诊断，还是“付费选品报告”？
- 免费结果给多少信息最合适，才能既有爽点又愿意付 0.5 元？
- 0.5 元解锁的完整报告应该包含哪些“明显值回票价”的内容？
- 用户分享后，别人看到卡片是否能马上理解这是一个有趣、私密但不尴尬的测试？

### 题库优化

- 当前 10 题是否仍然太像推荐问卷？
- 是否需要更多投射题，例如“你会选择哪种房间灯光/哪种音乐/哪种旅行收纳方式”？
- 是否需要降低直接装备相关问题的比例，增强趣味测试感？
- 是否需要给题目加入“反差”和“隐藏款”的爽点？
- 是否需要根据性别向、人群类型、关系状态动态出题？

### 画像命名

- 6 个主画像是否足够通俗？
- 是否需要每个画像有一个“星系副名”？
- 是否要避免 `直接点燃型` 这种略直接的表达？
- 隐藏路线是否应该从“无隐藏路线”改成“低存在感型”，增强用户获得感？

### 算法优化

- 是否需要增加维度分，例如：
  - 安全感
  - 隐私感
  - 氛围感
  - 直接反馈
  - 掌控感
  - 互动感
  - 隐藏/伪装需求
  - 便携收纳需求
- 是否要把主画像和隐藏路线拆成两个独立模型？
- 是否需要基于用户后续行为校准，例如重新推荐、收藏、点击详情、支付解锁？
- 是否需要建立人工标注测评集来评估画像稳定性和推荐准确性？

### 视觉与传播

- 应该做徽章、分享卡、人格证书，还是三者组合？
- 2.5D 徽章是否比 2D 更适合传播？
- 分享卡应该更像 MBTI 结果页，还是更像高级星系身份卡？
- 徽章图是否应该由 AI 生成素材，前端用 CSS/Canvas 叠加文字？
- 是否需要为每个主题生成不同风格的徽章/卡片？
