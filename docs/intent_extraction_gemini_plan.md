本方案基于前序已锁定的所有工程边界与业务需求输出。系统架构设计的核心目标是：在极其严苛地控制 Token 消耗与运行成本的前提下，实现高精准度、低延迟以及具备零信任数据隔离属性的情趣硬件自然语言推荐。

核心工程规约（已确立不可篡改的基准）
本系统的所有代码实现与后续迭代，必须绝对遵照以下已确认的规范执行：

采用 混合检索机制（Hybrid Search：标量精准过滤 + 向量语义召回）。

将检索召回的 Top-K 结果集直接作为卡片列表展示给用户，严禁大模型二次包装生成长文本。

前端界面接收 包含分页与权重的嵌套型数据结构（推荐工程标准）。

部署架构依托 Node.js + Prisma + Supabase，向量计算必须采用本地路径执行绝对隔离。

板块一：离线数据降维与基建层
本层负责彻底切除日常检索阶段对长文本的理解成本，将非结构化商品描述转化为高密度的结构化与向量化特征。

特征抽取（离线跑批）：在产品入库阶段，调用外部低成本 LLM API（如 DeepSeek），强制对长文本进行拆解，提取硬性指标（适用性别、噪音评级、波形频段等）并进行结构化打标。

向量聚合：将剔除冗余营销词汇的核心体感描述文本与标签合并，构建标准商品特征文本。

数据库映射：在 Supabase 中开启 pgvector 扩展。Prisma Schema 必须同时定义结构化字段与向量字段（如 semantic_vector Unsupported("vector(384)")），并建立 HNSW 索引以支撑毫秒级高并发查询。

板块二：零信任在线意图解析层
本层负责处理客户端发起的自然语言请求，核心原则是：极简 API 交互、最小权限数据流转与隐私物理隔离。

输入拦截与脱敏：前端传递的用户请求在进入网关时，仅保留与设备功能及生理偏好相关的文本，剥离任何可能指向身份特征的敏感数据。

NLU 单次解析：系统向云端 LLM 发送固定且携带强校验逻辑的 Prompt。要求模型仅进行自然语言理解任务，输出 JSON 字典，将请求强制切割为 hard_filters（如 {"gender": "female"}）与 soft_semantics（如 "强力吮吸，多波形"）。该环节为全链路唯一产生 Token 消耗的节点。

板块三：本地化计算与混合检索执行层
本层为系统核心枢纽，负责零代币开销的向量计算与数据库查询下推。

本地环境 Embedding计算：在 Node.js 服务内引入 onnxruntime-node，挂载量化版本的轻量级文本向量提取模型（如 all-MiniLM-L6-v2）。将解析出的 soft_semantics 直接在服务器内存中转化为稠密向量，彻底阻断敏感偏好向外部接口的泄露。

Prisma 原生 SQL 下推：通过 Prisma 的 $queryRaw 执行先过滤后检索的合并指令。

在 WHERE 子句中应用 hard_filters 进行绝对标量过滤，剔除底层属性不匹配的设备。

利用 <=> 运算符计算意图向量与库中 semantic_vector 的余弦相似度（Cosine Similarity），按 LIMIT K 倒序截断。

板块四：数据交付与前端渲染层
本层规范从服务端到客户端的数据流动形态，确保前端组件层能够高效、精准地承接检索结果。

脱水与嵌套规约：后端下发的 JSON 报文必须严格分为 meta（元数据）与 results（数据集）层级。彻底剔除庞大的高维向量数组与原始长文本。

分页与匹配度量化透出：

meta 节点携带 next_cursor 支撑前端列表视图的无限滚动加载逻辑。

results 节点内的每个对象必须显性下发 match_score。前端 React/Vue 组件依据此权重值，在产品展示卡片的显著位置渲染可视化的匹配度标识。

将解析出的用户意图数组透传，前端将据此在卡片标签上高亮显示触发召回的核心理由（如：“精准匹配：强力吮吸”），填补自然语言交互剥离后的解释力断层。
