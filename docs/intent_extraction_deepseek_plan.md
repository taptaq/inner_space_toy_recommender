二、详细模块设计
2.1 数据层设计（产品库）
每个产品存储为以下结构（示例）：

json
{
  "product_id": "toy_001",
  "name": "XX魔吮吸震动棒",
  "attributes": {
    "suction_intensity": 9,        // 吮吸强度 1-10
    "vibration_intensity": 7,
    "pattern_count": 20,           // 波形数量
    "noise_level_db": 45,          // 噪音分贝
    "gender": "female",
    "experience_level": "intermediate"
  },
  "description_text": "超强吮吸吸盘，20种波形，噪音仅45分贝，适合追求刺激的女生...",
  "embedding": [0.12, -0.34, ...]  // 预计算向量（768维）
}
低成本构建方式：

属性：手动整理（初期≤200款产品，2人天完成）

描述向量：使用免费多语言模型 paraphrase-multilingual-MiniLM-L12-v2（CPU即可运行），一次性生成所有产品的向量并存储。

2.2 自然语言理解模块（核心）
将用户输入转化为结构化条件，例如：

用户输入：
“我是女生，想找一个吮吸感更强一点的，波形更多的，噪音适中的。”

解析结果：

json
{
  "gender": "female",
  "suction_intensity": { "op": ">", "value": 7 },   // “更强” → >7（可配置阈值）
  "pattern_count": { "op": ">", "value": 12 },      // “更多” → >12
  "noise_level_db": { "op": "between", "min": 40, "max": 50 } // “适中” → 40~50dB
}
实现方案（按成本/准确度排序）：

方案	方法	成本	准确度	推荐阶段
A：纯规则	关键词词典 + 正则表达式	零	70%	初期MVP
B：轻量NER	用bert-base-chinese微调200条标注数据	0元（自标注）或100元云GPU	88%	稳定期
C：LLM API	调用deepseek-chat或gpt-3.5-turbo，prompt结构化输出	约0.001元/次	95%	处理模糊查询
推荐路径：

首周用方案A快速上线验证。

收集1000条真实用户查询，人工标注 → 微调方案B（耗时1天，成本0元）。

对未匹配或置信度低的查询，降级调用方案C（少量调用，成本可控）。

2.3 混合检索引擎
2.3.1 关键词检索（精确匹配）
使用 Elasticsearch 或轻量级 SQLite+BM25插件（如sqlite-bm25）。
对结构化属性建立索引，支持范围查询（>、<、between）。

查询示例（ES DSL）：

json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "gender": "female" } },
        { "range": { "suction_intensity": { "gt": 7 } } },
        { "range": { "pattern_count": { "gt": 12 } } },
        { "range": { "noise_level_db": { "gte": 40, "lte": 50 } } }
      ]
    }
  }
}
2.3.2 语义向量检索（模糊匹配）
将用户查询同样转换为向量（使用与产品相同的模型）。

使用 Faiss（CPU版）或 scikit-learn 的余弦相似度计算，召回与查询语义最相似的 Top-30 产品。

无需GPU，1万产品检索时间 < 50ms。

2.3.3 结果融合与重排序
采用 加权倒数融合（Weighted Reciprocal Rank Fusion, WRRF） 或简单线性加权：

python
final_score = α * keyword_score + β * semantic_score
# keyword_score：是否满足所有属性过滤条件（满足=1，不满足=0）
# semantic_score：余弦相似度（0~1）
推荐 α=0.6, β=0.4（属性精确优先，语义补充）。
排序后取 Top-5 作为最终推荐。

2.4 推荐理由生成（提升体验）
为用户提供简单易懂的推荐理由，增强信任感。
可选方案：

模板填充：例如“该产品吮吸强度9分，比您要求的更强；波形20种，噪音45dB适中。”

LLM生成（低成本）：调用 LLM API 根据产品和查询生成一句话推荐语（约0.0005元/次）。

四、实施路线图（分4个迭代）
迭代1（第1-3天）：MVP上线
手工整理50款产品的属性标签和描述。

实现规则型NLU（词典+正则）。

仅使用关键词检索（无向量）。

完成一个简单的命令行或Streamlit界面。

目标：验证流程，能给出基本合理推荐。

迭代2（第4-7天）：加入语义检索
对所有产品生成Embedding向量。

集成Faiss向量检索。

实现加权融合排序。

测试10个真实查询，调整权重。

目标：推荐准确度提升，能够理解近义表达（如“安静”匹配“噪音低”）。

迭代3（第2周）：训练轻量NER模型（可选）
从用户日志中收集200条查询+人工标注结构。

微调bert-base-chinese（租用GPU 1小时约5元，或免费Colab）。

替换规则NLU。

目标：支持更复杂的复合句、否定句（如“不要太响”）。

迭代4（第3周以后）：引入LLM处理长尾（可选）
设定规则：当NER模型置信度<0.7或查询长度>30字时，调用LLM API。

缓存结果（相同查询1小时内不重复调用）。

目标：覆盖几乎所有自然语言变体，准确度接近人类客服。

六、准确度保障与评价方法
6.1 离线评估（推荐）
构建测试集：50条“查询→期望产品ID”标注。

指标：Recall@5（推荐列表中包含期望产品的比例），MRR。

迭代中持续优化。

6.2 在线A/B测试
对比纯关键词、混合检索、混合+LLM三种模式下的用户点击率（或购买转化率）。

6.3 规则库持续优化
维护同义词词典（如“吮吸感”→“suction”，“震动强”→“vibration_high”）。

定期根据bad case更新映射逻辑。

七、总结
本设计方案提供了一个低成本、可渐进优化的完整路径：

核心思想：混合检索（BM25+向量） 比纯向量成本更低且更可控；规则与AI结合，把复杂查询交给小模型或API，避免过度工程。

最快落地：3天内上线MVP（规则+关键词），1周内加入向量，准确度可达85%以上。

长期扩展：当数据积累到万级别，可升级为深度排序模型（如双塔DSSM），但当前方案已足够满足初期业务需求。
